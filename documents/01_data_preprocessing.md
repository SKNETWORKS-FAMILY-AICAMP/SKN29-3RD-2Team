# 데이터 수집 및 전처리

## 1. 데이터 개요

### 1-1. 데이터 출처

본 프로젝트는 세 가지 출처의 법률 데이터를 수집하여 RAG 파이프라인에 활용한다. 질의회시는 전처리까지 완료하였으나 검색 파이프라인에는 통합하지 않았다(단, 시스템 평가 데이터셋으로는 활용함 - 04_test_plan_and_results.md 참조).

| 데이터 | 출처 | 형식 | 사용 여부 |
|--------|------|------|-----------|
| 법령 | 국가법령정보센터 (law.go.kr) | PDF | ✅ 사용 |
| 판례 | 대법원 종합법률정보 (glaw.scourt.go.kr) | Markdown | ✅ 사용 |
| 질의회시 | 고용노동부 질의회시집 | PDF | ⛔ 검색 미사용 (평가 데이터로 활용) |

### 1-2. 청킹 전략

데이터 유형별로 최적화된 청킹 전략을 적용하여 검색 효율을 극대화하였다.

| 데이터 | 청킹 전략 | 청킹 단위 |
|--------|-----------|-----------|
| 법령 | 구조 기반 청킹 | 조문(條) 단위 |
| 판례 | SAC (Summary-Augmented Chunking) | 판례 1건 → search / brief 2개 Document |
| 질의회시 | 질문 단위 청킹 | 질의(question) 단위 |

## 2. 전처리 실행

### 2-1. 통합 실행 스크립트

모든 전처리는 `run_preprocess.py`를 통해 순차적으로 실행된다.

```bash
python -m backend.preprocess.run_preprocess
```

**실행 순서**

```
run_case()   → data/raw/case → data/process/case
run_law()    → data/raw/law  → data/process/law
run_sac()    → data/process/case → data/cache/sac
```

`run_qna()`는 코드에 존재하나 현재 주석 처리되어 실행에서 제외되어 있다.

### 2-2. 데이터 디렉토리 구조

전처리 완료 후 데이터는 다음 구조로 저장된다.

```
data/
├── raw/
│   ├── law/         # 법령 PDF
│   ├── case/        # 판례 Markdown
│   └── pdf/         # 질의회시 PDF (미사용)
├── process/
│   ├── law/         # 법령 JSON
│   ├── case/        # 판례 JSON
│   └── qna/         # 질의회시 JSON (미사용)
└── cache/
    └── sac/         # 판례 SAC 캐시 JSON
```

## 3. 법령 전처리 (preprocess_law.py)

### 3-1. 수집

| 항목 | 내용 |
|------|------|
| **출처** | 국가법령정보센터 (law.go.kr) |
| **대상** | 근로기준법 등 노동 관련 법령 PDF |
| **저장 경로** | `data/raw/law/*.pdf` |

### 3-2. 청킹 전략 — 구조 기반 청킹

조문은 법률 해석의 최소 독립 단위이며, 조문 간 의미 경계가 명확하다. 고정 길이 청킹 대신 **제X조** 단위로 분리함으로써 조문 경계를 넘는 청크 생성을 방지하고 검색 정밀도를 높인다. 장(章) / 절(節) / 조(條) 계층 정보를 메타데이터로 함께 저장하여 ChromaDB flat metadata 구조에 최적화하였다.

### 3-3. 전처리 상세

법령 PDF를 **조문(條) 단위**로 분리하여 Document를 생성한다. ChromaDB에 저장하기 위해 flat한 메타데이터 구조를 사용한다.

**파싱 흐름**

```
PDF 텍스트 추출 (PyMuPDF)
    ↓
페이지 단위 분리
    ↓
노이즈 제거 (헤더, 페이지 번호, 법제처 워터마크, 개정이력 태그 등)
    ↓
장(章) / 절(節) / 조문(條) 단위 구조 파악
    ↓
조문별 Document 생성
    ↓
data/process/law/**/*.json 저장
```

**노이즈 제거 대상**

- `<개정 2021. 1. 5.>` 형태의 꺾쇠 개정이력 태그
- `[본조신설 2024. 10. 22.]`, `[시행일: 2027. 1. 1.]` 형태의 대괄호 메타 태그
- 페이지 말미에 잔류하는 조문 번호 (`근로기준법 제43조의6` 등)
- 법제처, 국가법령정보센터, 페이지 번호 단독 라인

**Document 구조**

```json
{
  "page_content": "제23조(해고 등의 제한) ① 사용자는 근로자에게 ...",
  "metadata": {
    "law_name": "근로기준법",
    "chapter_no": "제2장",
    "chapter_title": "근로계약",
    "section_no": "제1절",
    "section_title": "...",
    "article_no": "제23조",
    "article_title": "해고 등의 제한"
  }
}
```

- `section_no` / `section_title`은 해당 조문에 절(節) 구분이 있는 경우에만 포함된다.
- `page_content`는 `조문번호(제목) 본문` 형식으로 구성하여 임베딩 대상으로 사용한다.

## 4. 판례 전처리 (preprocess_case.py)

### 4-1. 수집

| 항목 | 내용 |
|------|------|
| **출처** | 대법원 종합법률정보 (glaw.scourt.go.kr) |
| **대상** | 노동 관련 주요 대법원 판례 |
| **형식** | Markdown (`.md`) |
| **저장 경로** | `data/raw/case/{카테고리}/*.md` |

**수집 카테고리 (대분류)**

| 카테고리 | 설명 |
|----------|------|
| 임금의 정의범위 | 통상임금, 평균임금, 성과급 등 |
| 근로시간 | 포괄임금제, 연장·야간·휴일근로 등 |
| 퇴직 및 해고 | 부당해고, 해고예고, 정리해고 등 |
| 근로계약 | 근로자성 판단, 기간제 계약 등 |
| 보험 및 산재 | 업무상 재해 인정 기준 등 |
| 직장 내 권리 | 직장 내 괴롭힘, 성희롱 등 |

### 4-2. 청킹 전략 — SAC (Summary-Augmented Chunking)

판례 전문은 수천 자에 달해 그대로 임베딩하면 **Vocabulary Mismatch** 문제가 발생한다. 사용자의 구어체 질의와 법률 문어체 판례 사이의 어휘 불일치로 인해 의미적으로 관련된 판례가 검색에서 누락될 수 있다.

이를 해결하기 위해 `preprocess_sac.py`에서 판례 1건을 두 개의 Document로 분리한다.

| 필드 | 설명 | 용도 |
|------|------|------|
| `search` | 사용자 질의 어휘에 가까운 구어체 요약 | 임베딩 → 유사도 검색 |
| `brief` | 판시사항·판결요지 중심의 법률 요약 | LLM 답변 생성 컨텍스트 |

**search 생성 전략**

`search`는 임베딩 대상이므로 사용자 질의 어휘와 최대한 가깝게 작성되어야 한다. LLM 프롬프트에서 `[핵심 쟁점]`과 `[판결 결론]` 두 항목만 생성하도록 제약하며, category를 중심으로 작성하도록 명시한다. 본문에 여러 법률 문제가 등장하더라도 category와 직접 관련된 쟁점만 우선 기술한다.

**brief 생성 전략**

`brief`는 LLM이 답변을 생성할 때 참조하는 컨텍스트다. `[사건 개요]`, `[판결 요지]`, `[핵심 법리]`, `[참조 조문]` 네 항목으로 구성하여 법률 용어 중심의 구조화된 요약을 생성한다.

**카테고리별 차등 적용**

프롬프트에 `{category}`를 명시적으로 주입하여 카테고리(임금, 근로시간, 해고 등)별로 핵심 쟁점의 초점이 달라지도록 설계하였다. 동일한 판례라도 카테고리에 따라 강조되는 법적 쟁점이 달라지므로, category를 프롬프트 규칙의 기준으로 사용한다.

### 4-3. 전처리 상세

Markdown 파일 1개 = 판례 1건으로 처리한다. 별도 파싱 없이 파일 전체를 `page_content`로 저장하고, 폴더명에서 카테고리를 추출한다.

**파싱 흐름**

```
data/raw/case/{카테고리}/*.md 탐색
    ↓
파일 전체 텍스트 → page_content
폴더명 → category (앞의 "1." 같은 번호 제거 후 추출)
    ↓
data/process/case/{카테고리}/{사건번호}.json 저장
```

**카테고리 정규화**

폴더명이 `1.임금의 정의범위` 형태인 경우, `.` 이후를 추출하여 `임금의 정의범위`로 저장한다.

**SAC 처리 흐름**

```
data/process/case/*.json
    ↓
판례 전체 page_content 결합
    ↓
category별 dual summary 생성 (LLM 호출)
    ↓
data/cache/sac/{사건번호}.json 저장
```

이미 처리된 사건번호는 캐시 파일이 존재하면 건너뛴다. (`--force` 옵션으로 재생성 가능)

**Document 구조**

```json
{
  "page_content": "# 대법원 2011. 7. 14. 선고 2011다23149 판결 ...\n\n## 판시사항\n...",
  "metadata": {
    "source_file": "2011다23149.md",
    "category": "임금의 정의범위"
  }
}
```

**SAC 프롬프트 구조**

LLM에 판례 전문과 카테고리를 입력하여 `<search>`, `<brief>` 두 섹션을 동시에 생성한다. 태그 외 텍스트 출력을 금지하여 파싱 안정성을 확보하였다.

```
아래 판례를 두 섹션으로 요약하라.
출력은 반드시 <search>태그와 <brief>태그만으로 구성하라. 태그 외 텍스트 금지.

<precedent>
{content}
</precedent>

<category>
{category}
</category>

중요 규칙
1. category는 이 판례의 대표 법률 쟁점이다.
2. [핵심 쟁점]은 반드시 category를 중심으로 작성한다.
3. 본문에 여러 법률 문제가 등장하더라도 category와 직접 관련된 쟁점을 우선한다.
4. category와 무관한 일반적 법리를 핵심 쟁점으로 작성하지 마라.

<search>
[핵심 쟁점] category({category})와 관련된 핵심 법적 쟁점 1~2문장.
[판결 결론] category({category})에 대한 법원의 판단 1문장.
</search>

<brief>
[사건 개요] 발생한 분쟁을 법률 용어로 2문장.
[판결 요지] 법원의 핵심 판단 2~3문장.
[핵심 법리] 이 판례에서 확립된 법적 원칙 1~2문장.
[참조 조문] 적용된 법령명과 조문번호. 예: 근로기준법 제56조
</brief>
```

**SAC 출력 예시** (`2006도777` — 근로자성 판단)

```json
{
  "case_no": "2006도777",
  "metadata": {
    "source_file": "2006도777.md",
    "category": "근로자성 판단"
  },
  "search": "[핵심 쟁점]\n미용학원 강사가 계약서상 위촉·사업소득으로 처리되었더라도, 근로기준법상 '근로자'로 볼 수 있는지 여부(실질적 종속관계 판단 기준). \n또한 기본급 고정 여부, 원천징수(4대보험 미가입) 등 외형적 사정만으로 근로자성을 부정할 수 있는지가 문제이다.\n\n[판결 결론]\n미용학원 강사는 임금을 목적으로 사용자에게 종속되어 근로를 제공한 근로자에 해당한다.",
  "brief": "[사건 개요]\n미용학원 운영자와 피부관리 강사 사이에 강사위촉계약/입사계약이 체결되었고, 학원은 사업소득세 처리 및 4대보험 미가입 등으로 강사의 근로자성을 부정하였다.\n강사가 근로기준법상 근로자임을 전제로 권리주장을 하며 다툼이 발생하였다.\n\n[판결 요지]\n대법원은 근로자성은 계약 형식보다 실질에 따라, 업무 내용의 사용자 지정·지휘감독, 근무시간·장소 지정 및 구속, 대체(대행) 가능성 제한, 보수의 근로대상성 및 계속적 제공 여부 등 종합 사정을 보아 판단해야 한다고 보았다.\n해당 강사는 강의종목·시간·장소가 지정되고 출·퇴근 관리가 이루어지며, 폐강 예외를 제하면 수강생 수에 따라 보수가 변동되지 않고, 대리강의가 제한되는 등 종속적 관계가 인정되어 근로자에 해당한다고 판단하였다.\n\n[핵심 법리]\n근로자성 판단은 계약의 명칭(고용/도급, 위촉/용역 등)보다 임금을 목적으로 종속적인 관계에서 사용자에게 근로를 제공했는지 여부에 따라야 한다.\n기본급 고정 여부, 원천징수 형태, 4대보험 가입 여부 등은 사용자의 경제적 우월성이 개입될 여지가 있어 그 사정만으로 근로자성을 쉽게 부정할 수 없다.\n\n[참조 조문]\n구 근로기준법(2005. 3. 31. 법률 제7465호로 개정되기 전의 것) 제14조"
}
```

## 5. 질의회시 전처리 (preprocess_qna.py) — 미사용

> 본 프로젝트에서는 질의회시 데이터를 RAG 파이프라인에 포함하지 않았다. 전처리 코드 및 파싱 결과(`qna.json`)는 구현 완료 상태이며, 향후 확장 시 벡터스토어에 추가 적재 가능하다.

### 5-1. 수집

| 항목 | 내용 |
|------|------|
| **출처** | 고용노동부 「근로기준법 질의회시집 2018.4~2023.6」 |
| **형식** | PDF → JSON 변환 |
| **저장 경로** | `data/process/qna/qna.json` |

### 5-2. 청킹 전략 — 질문 단위 청킹

질의(question)만 `page_content`로 분리하여 임베딩한다. 사용자의 비정형 사례 설명이 행정해석 질의와 직접 매칭되도록 설계하였으며, 회시(answer) 전문은 metadata에 저장하여 검색 후 꺼내 쓰는 구조이다.

### 5-3. 전처리 상세

**Document 구조**

```json
{
  "page_content": "질의: 「군형법」상 군인이 근로기준법상 근로자인지",
  "metadata": {
    "title": "「군형법」상 군인의 근로자성 여부",
    "answer": "국가 또는 지방공무원도 임금을 목적으로 ...",
    "chapter_num": 1,
    "chapter_name": "총칙",
    "reference": "(근로기준정책과-1384, 2021.5.11.)",
    "ref_date": "2021-05-11",
    "start_page": 25,
    "source": "근로기준법 질의회시집 2018.4~2023.6",
    "title_keywords": "「군형법」상 군인의 근로자성"
  }
}
```

> 질의회시 데이터는 전처리 완료 상태로 유지되며, 향후 RAG 파이프라인 확장 시 ChromaDB에 추가 적재하여 활용할 수 있다.

## 6. 임베딩 전략

### 6-1. 임베딩 모델

모든 데이터 유형에 대해 동일한 임베딩 모델을 사용한다.

| 항목 | 내용 |
|------|------|
| **임베딩 모델** | OpenAI `text-embedding-3-small` |
| **설정 위치** | `backend/config.py` |
| **적용 DB** | ChromaDB (laws / precedents / qna) |

### 6-2. 데이터 유형별 임베딩 입력 텍스트

각 데이터 유형이 ChromaDB에 저장될 때 `page_content`로 사용되는 텍스트가 다르다. 이 텍스트가 곧 임베딩 벡터로 변환되는 대상이다.

| 데이터 | page_content 구성 | 근거 |
|-------|------------------|------|
| **법령** | `{장 제목}\n\n{조문 제목}\n\n{조문 본문}` | 제목과 본문을 함께 임베딩하여 검색 정확도 향상, 공백 정규화 적용(`" ".join(text.split())`) |
| **판례** | SAC `search` 필드 (LLM 생성 구어체 요약) | 법률 문어체인 원문 대신 구어체 요약을 임베딩하여 사용자 질의와의 어휘 간극(Vocabulary Mismatch) 해소 |
| **질의회시** | `"질의: {질문 내용}"` | 질문 단위 검색을 위해 질문 텍스트만 임베딩, 회시(answer)는 metadata에 별도 보관 |

### 6-3. 검색 및 재순위화

ChromaDB의 `similarity_search`로 1차 후보를 추출한 후, Cross-Encoder 기반 Reranker로 재순위화한다.

| 단계 | 도구 | 상세 |
|------|------|------|
| 1차 검색 | ChromaDB `similarity_search` | `k=30` (판례), `k=7` (법령, 하이브리드 스코어링) |
| 2차 재순위화 | `Dongjin-kr/ko-reranker` (CrossEncoder) | 30개 → 상위 5개 압축, (질문, 판례 brief) 쌍 단위 점수 계산 |
| 법령 하이브리드 | metadata 정확 매칭(1.0) + 질의 유사도(0.6) + 임베딩 유사도(0.3) | 법령 검색 정밀도와 재현율 균형 확보 |

## 7. 전처리 품질 관리

### 7-1. 중복성(Duplication) 관리

중복 제거는 주로 검색 결과 수준에서 처리되며, 데이터셋 전체를 대상으로 한 사전 중복 검사는 수행하지 않는다.

| 단계 | 중복 제거 기준 | 코드 위치 |
|------|---------------|-----------|
| 판례 검색 결과 | 사건번호(case_no) 기준 `seen` 집합 | `backend/nodes/retrieval.py` |
| SAC 캐시 로딩 | 사건번호 기준 `seen_cases` 집합 | `backend/builders/precedent_builder.py` |
| 법령 검색 결과 | `(law_name, article_no)` 기준 `seen` 집합 | `backend/retrievers/law_retriever.py` |
| 법률 PDF 파싱 | 동일 페이지 번호 재파싱 방지 | `backend/preprocess/preprocess_law.py` |

### 7-2. 결측치(Missing Value) 처리

전처리 파이프라인(pdf/md → json) 단계에서는 별도의 결측치 검사를 수행하지 않는다.
결측치 처리는 근로계약서 OCR 모듈에서 부분적으로 처리된다.

| 처리 대상 | 방식 | 코드 위치 |
|-----------|------|-----------|
| OCR 필드 null 체크 | `if not v or v.lower() == "null"` | `ocr_contract/rules/core/blank_checker.py` |
| 필수 기재사항 누락 | `REQUIRED_FIELDS` 사전 기반 검사 | `ocr_contract/rules/validators/required.py` |
| 결측 시각화 | ✅ (정상) / ❌ (누락) 아이콘 표시 | `ocr_contract/pipeline.py` |

### 7-3. 편향성(Bias) 고려 사항

현재 별도의 편향성 분석 도구는 도입되어 있지 않다. 다만 데이터 수집 단계에서 다음과 같은 불균형이 존재하므로, 검색 및 답변 생성 시 이 점을 고려해야 한다.

- **법률별 조문 수 불균형**: 산업안전보건법(572조)이 기간제법(41조)보다 약 14배 많은 조문을 보유 — 검색 결과에서 산업안전보건법이 과대 대표될 가능성이 있음
- **판례 카테고리 불균형**: 일부 카테고리에 판례가 집중되어 있을 가능성
- **데이터 불균형 완화 장치**: Cross-Encoder Reranker(유사도 기반 재순위화)와 하이브리드 스코어링을 통해 순수 문헌 수에 의존하지 않도록 설계

> 편향성 정밀 분석은 향후 과제로 남겨둔다.

## 8. 적용 기술 검토

### 8-1. 형태소 분석

본 프로젝트는 KoNLPy 등 형태소 분석기를 사용하지 않는다. 그 이유는 다음과 같다.

1. **법률 텍스트의 특성**: 법령과 판례는 정형화된 문어체로, 조문 단위 청킹으로 충분한 의미 단위가 확보된다.
2. **임베딩 모델의 대체**: OpenAI `text-embedding-3-small`이 어휘 수준의 의미 표현을 처리하므로, 별도의 형태소 분석 없이도 유사도 검색이 가능하다.
3. **SAC 요약의 역할**: 판례의 경우 SAC(Summary-Augmented Chunking)을 통해 LLM이 직접 구어체 요약을 생성하므로, 형태소 분석이 필요한 어휘 변환을 LLM이 대체한다.

다만 질의회시 전처리에서는 제목 키워드 추출을 위해 간단한 불용어 처리를 수행한다.

```python
# preprocess_qna.py — str.split() 기반 불용어 필터링 (형태소 분석 비사용)
stopwords = {"여부", "관련", "경우", "대한", "에서", "있는", "하는", "으로", "에"}
words = title.split()
keywords = [w for w in words if w not in stopwords and len(w) > 1]
```

### 8-2. Graph DB

본 프로젝트는 Neo4j 등의 그래프 데이터베이스(Graph DB)를 사용하지 않는다. 코드베이스에서 `graph`라는 용어는 모두 **LangGraph**(LLM 워크플로우 오케스트레이션 프레임워크)를 의미한다.

| 파일 | 역할 |
|------|------|
| `backend/graph.py` | LangGraph StateGraph — RAG 파이프라인의 4개 노드 순차 실행 |
| `backend/supervisor/graph.py` | LangGraph SupervisorGraph — Multi-Agent 오케스트레이션 |
| `backend/calculator/graph.py` | LangGraph ReAct Agent — 퇴직금 계산기 |
| `backend/nodes/graph_state.py` | LangGraph State 타입 정의 (14개 필드) |

따라서 청킹 전략은 Vector DB(ChromaDB) 입력만을 고려하여 설계되었으며, Knowledge Graph 구축은 향후 과제로 남겨둔다.
